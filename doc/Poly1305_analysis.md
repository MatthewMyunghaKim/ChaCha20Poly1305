## 1. Poly1305 알고리즘 수학적 배경

### 1.1 기본 정의

Poly1305는 Daniel J. Bernstein이 설계한 일회용(One-time) 메시지 인증 코드(MAC)이다. 수학적으로 다음과 같이 정의된다:

```
tag = ((c₁ × r^n + c₂ × r^(n-1) + ... + cₙ × r¹) mod p) + s
```

여기서:
- **p = 2^130 - 5** (메르센 소수에 가까운 소수)
- **r:** 128비트 비밀 키 (클램핑 적용 후 약 106비트 유효)
- **s:** 128비트 일회용 패드 (ChaCha20 블록 0에서 생성)
- **cᵢ:** 16바이트 메시지 블록에 0x01 바이트를 추가한 129비트 값
- **n:** 메시지 블록 수 (최대 93, RFC 7539 제한)

### 1.2 호너 방식 vs 직접 다항식 평가

**소프트웨어 구현(호너 방식):**

```
acc = 0
for each block cᵢ:
    acc = (acc + cᵢ) × r mod p
tag = acc + s
```

이 방식은 **순차적**이다. 각 블록을 처리하기 전에 이전 블록의 결과가 필요하므로, n개 블록에 대해 n번의 곱셈을 직렬로 수행해야 한다.

**하드웨어 구현(직접 다항식 평가):**

```
tag = (c₁ × r^n + c₂ × r^(n-1) + ... + cₙ × r¹) mod p + s
```

이 방식은 각 블록을 **독립적으로** r의 적절한 거듭제곱과 곱할 수 있다. 하드웨어 파이프라인에서는 매 클럭마다 새로운 블록을 수용하면서도, 각 블록에 대한 곱셈을 파이프라인으로 처리할 수 있다.

### 1.3 하드웨어에서 직접 평가를 선택한 이유

**핵심 문제:** 호너 방식에서는 `(acc + cᵢ) × r`의 결과가 나와야 다음 블록을 처리할 수 있다. 136비트 모듈러 곱셈은 12~18클럭이 소요되므로, 매 클럭마다 새 블록을 처리할 수 없다.

**해결책:** 직접 다항식 평가 방식을 사용하면:
1. 각 블록 cᵢ에 대해 r^(n-i+1)을 **사전 계산**
2. 매 클럭마다 새 블록을 파이프라인에 투입
3. cᵢ × r^(n-i+1)의 결과를 누적기에 가산
4. **처리량: 128비트/클럭** (정상 상태)

## 2. 왜 r^n 사전 계산이 필요한가?

### 2.1 근본적인 동기

본 설계에서 r^n 사전 계산은 **파이프라인 처리량을 극대화**하기 위한 핵심 설계 결정이다.

#### 2.1.1 호너 방식의 한계 (소프트웨어 접근)

```
소프트웨어 방식:
  acc = 0
  for i = 1 to n:
      acc = (acc + c[i]) × r mod p     ← 순차 의존성!

타이밍:
  블록 1: acc₁ = (0 + c₁) × r         [18클럭 소요]
  블록 2: acc₂ = (acc₁ + c₂) × r      [acc₁이 나올 때까지 대기!]
  블록 3: acc₃ = (acc₂ + c₃) × r      [acc₂가 나올 때까지 대기!]
  ...
  총 시간: n × 18클럭 = 순차적
```

18클럭짜리 곱셈기를 사용하면, 10블록 메시지에 180클럭이 필요하다. **매 클럭마다 새 블록을 처리할 수 없다.**

#### 2.1.2 r^n 사전 계산 방식의 장점

```
사전 계산 방식:
  r^n은 첫 블록 수신 시 계산 시작

  블록 1: c₁ × r^n     [파이프라인에 투입]
  블록 2: c₂ × r^(n-1) [다음 클럭에 투입]  ← 이전 결과 불필요!
  블록 3: c₃ × r^(n-2) [다음 클럭에 투입]  ← 이전 결과 불필요!
  ...

  모든 곱셈 결과를 누적기에 가산

  총 시간: 파이프라인 깊이 + n클럭
```

**핵심 차이:** 각 블록의 처리가 **이전 블록의 결과에 의존하지 않는다.** 따라서 매 클럭마다 새 블록을 파이프라인에 투입할 수 있어, 정상 상태 처리량이 **128비트/클럭**에 달한다.

### 2.2 r^n이 매 블록마다 달라지는 이유

첫 번째 블록 c₁은 r^n과 곱하고, 두 번째 블록 c₂는 r^(n-1)과 곱해야 한다. 이는 다항식 평가의 본질이다:

```
tag = c₁ × r^n + c₂ × r^(n-1) + ... + cₙ × r¹
```

따라서 r^n 계산 모듈(`r_power_n`)은:
1. 첫 블록 수신 시 n 값과 r 값을 캡처
2. n에서 시작하여 매 블록마다 지수를 1씩 감소
3. 해당하는 r^(현재 지수)를 파이프라인 곱셈기에 공급

### 2.3 이진 거듭제곱법 선택 이유

r^n을 계산하는 가장 단순한 방법은 r을 n번 곱하는 것이나, 이는 n번의 순차 곱셈이 필요하다:

```
단순 방법: r → r² → r³ → ... → r^n  (n-1번 순차 곱셈)
n=93일 때: 92 × 18클럭 = 1,656클럭
```

**이진 거듭제곱법:**

```
사전 계산: r, r², r⁴, r⁸, r¹⁶, r³², r⁶⁴  (6번 제곱, 병렬 가능!)

n = Σ nᵢ × 2^i 이므로:
r^n = r^(n₆×64) × r^(n₅×32) × r^(n₄×16) × r^(n₃×8) × r^(n₂×4) × r^(n₁×2) × r^(n₀×1)

최대 6번의 곱셈만 필요! (순차가 아닌 3단 트리 구조)
```

예를 들어, n=93 = 1011101₂:
```
r^93 = r^64 × r^16 × r^8 × r^4 × r^1
     = r^(1×64 + 0×32 + 1×16 + 1×8 + 1×4 + 0×2 + 1×1)
```

### 2.4 고정 지연의 중요성

하드웨어 파이프라인에서는 **지연이 데이터에 의존하지 않아야** 한다. n의 값에 관계없이 r^n 계산은 항상 동일한 클럭 수가 소요되어야 한다. 이진 거듭제곱법에서:

- r^(2^i) 사전 계산: 6개 곱셈기가 **병렬로** 동시 실행
- 최종 조합: 3단 캐스케이드 곱셈 (`mul_red_pipeline`)
- n의 비트가 0이면 해당 위치에 1을 곱함 (항등원)
- **n 값에 관계없이 동일한 파이프라인 지연**

### 3.1 r_power_n.vhd (기본 구현) 구조

```
               │  │                                              │ │

               │     │  u2: C×D (r⁸ or 1 × r⁴ or 1)       │   │ │
               │     │  u3: E×F (r² or 1 × r¹ or 1)        │   │ │
               │     │          ↓ 병렬                       │   │ │
               │     │  u4: u1_out × u2_out                 │   │ │
               │     │          ↓                            │   │ │
               │     │  u5: u4_out × u3_out (지연 정렬)     │   │ │
               │     │          ↓                            │   │ │

               │     └──────────────────────────────────────┘   │ │
               └─────────────────────────────────────────────────┘
```

### 3.2 6개 거듭제곱 사전 계산기 (병렬)

```vhdl
-- VHDL Generate 문으로 6개의 mul136_mod_red를 인스턴스화
GEN1: for i in 0 to 5 generate
    first_one: if i = 0 generate
        u1: mul136_mod_red port map(clk, r_in_padded, r_in_padded, mod_red_out(0));
        -- r² = r × r
    end generate;

    others: if i > 0 generate
        u2: mul136_mod_red port map(clk, mod_red_out(i-1)_padded,
                                         mod_red_out(i-1)_padded, mod_red_out(i));
        -- r^(2^(i+1)) = r^(2^i) × r^(2^i)
    end generate;
end generate;
```

**주의:** 6개의 곱셈기가 **직렬 체인**으로 연결된다:
- `mod_red_out(0)` = r² (r_in × r_in의 결과)
- `mod_red_out(1)` = r⁴ (mod_red_out(0) × mod_red_out(0))
- `mod_red_out(2)` = r⁸ ...
- ...
- `mod_red_out(5)` = r⁶⁴

그러나 각 곱셈기 내부의 mul_144는 4개의 mul_72를 **병렬로** 사용한다. 즉 6개의 mul136_mod_red가 직렬 체인이지만, 각각 내부에서 64개의 mult_gen_0을 병렬 사용한다.

### 3.3 시프트 레지스터 지연 구조

각 r^(2^i) 값이 최종 조합 단계에 동시에 도착하도록 시프트 레지스터로 지연을 보정한다:

```
r⁶⁴: mul136_mod_red[5]에서 직접 출력 → 지연 0  (가장 마지막에 계산됨)
r³²: 시프트 레지스터 18단 지연 → r⁶⁴ 완료 시점에 맞춤
r¹⁶: 시프트 레지스터 36단 지연
r⁸:  시프트 레지스터 54단 지연
r⁴:  시프트 레지스터 72단 지연
r²:  시프트 레지스터 90단 지연
r¹:  시프트 레지스터 108단 지연 (가장 먼저 사용 가능하지만 가장 오래 대기)
n:   시프트 레지스터 108단 지연 (r¹과 동일 타이밍)
```

**패턴:** 각 단계마다 18클럭씩 차이 = mul136_mod_red 하나의 지연

### 3.4 이진 선택 로직

n_val의 각 비트에 따라 해당 거듭제곱 또는 항등원(1)을 선택:

```vhdl
-- n_val(6): 최상위 비트
if n_val(6) = '1' then
    pipe_A_in <= "000000" & r64;     -- r⁶⁴ 선택
else
    pipe_A_in <= "000000" & r32;     -- r³² 선택 (n_val(6)=0이면 A에 r³² 전달)
end if;

-- n_val(5)
if n_val(5) = '1' then
    pipe_B_in <= "000000" & r16;     -- r¹⁶ 선택
else
    pipe_B_in <= x"000...001";       -- 1 (항등원)
end if;

-- n_val(4) ~ n_val(1): 동일한 패턴
-- n_val(4) → pipe_C_in: r⁸ or 1
-- n_val(3) → pipe_D_in: r⁴ or 1
-- n_val(2) → pipe_E_in: r² or 1
-- n_val(1) → pipe_F_in: r¹ or 1
```

**특이점:** n_val(6)이 0일 때 pipe_A_in에 r³²를 넣는다. 이는 n_val(5)가 1이면 pipe_B_in에 r¹⁶이 가고, A×B = r³² × r¹⁶ = r⁴⁸이 되어 올바른 결과를 보장하기 위한 최적화이다.

### 3.5 mul_red_pipeline의 3단 캐스케이드

```
시간 →    |--- 18클럭 ---|--- 18클럭 ---|--- 18클럭 ---|

Level 1 (u1, u2, u3 동시 시작):
  u1: pipe_A × pipe_B → u1_out (r^{a+b})
  u2: pipe_C × pipe_D → u2_out (r^{c+d})
  u3: pipe_E × pipe_F → u3_out (r^{e+f}) → Shift_reg_u3[0~17]

Level 2 (Level 1 완료 후):
  u4: u1_out × u2_out → u4_out (r^{a+b+c+d})
      u3_out는 18단 시프트 레지스터에서 대기

Level 3 (Level 2 완료 후):
  u5: u4_out × Shift_reg_u3[17] → data_out (r^{a+b+c+d+e+f} = r^n)
```

**총 mul_red_pipeline 지연:**
- Level 1: 18클럭 (mul136_mod_red)
- Level 2: 18클럭
- Level 3: 18클럭
- **총: 54클럭** (3단 × 18클럭/단)

## 4. 곱셈기 계층 구조와 병렬성

### 4.1 기본 구현(src/) 곱셈기 계층

```
mul136_mod_red (136×136 → 130비트)
  ├── mul_144 (136×136 → 272비트, 12스테이지)
  │     ├── mul_72[0] (68×68 → 136비트, 9스테이지)  ← A₀×B₀
  │     │     ├── mul_36[0] (34×34 → 68비트, 6스테이지)  ← A₀₀×B₀₀
  │     │     │     ├── mult_gen_0[0] (17×17 → 34비트, 3스테이지)
  │     │     │     ├── mult_gen_0[1] (17×17)
  │     │     │     ├── mult_gen_0[2] (17×17)
  │     │     │     └── mult_gen_0[3] (17×17)
  │     │     ├── mul_36[1]  ← A₀₀×B₀₁
  │     │     │     └── mult_gen_0 × 4
  │     │     ├── mul_36[2]  ← A₀₁×B₀₀
  │     │     │     └── mult_gen_0 × 4
  │     │     └── mul_36[3]  ← A₀₁×B₀₁
  │     │           └── mult_gen_0 × 4
  │     ├── mul_72[1]  ← A₀×B₁
  │     │     └── (동일 구조: mul_36 × 4, mult_gen_0 × 16)
  │     ├── mul_72[2]  ← A₁×B₀
  │     │     └── (동일 구조)
  │     └── mul_72[3]  ← A₁×B₁
  │           └── (동일 구조)
  │
  └── mod_red_1305 (261비트 → 130비트, 6스테이지)
```

#### 4.1.1 병렬 곱셈기 수 (기본 구현, 단일 mul136_mod_red당)

| 레벨 | 모듈 | 인스턴스 수/부모 | 총 병렬 인스턴스 |
|------|------|-----------------|------------------|
| 1 | mult_gen_0 (17×17) | 4/mul_36 | **64** |
| 2 | mul_36 (34×34) | 4/mul_72 | **16** |
| 3 | mul_72 (68×68) | 4/mul_144 | **4** |
| 4 | mul_144 (136×136) | 1/mul136_mod_red | **1** |

**한 번의 136×136 곱셈에 64개의 17×17 곱셈기가 동시에 작동한다.**

### 4.2 카라추바 최적화(src_dsp_opt/) 곱셈기 계층

```
mul136_mod_red (136×136 → 130비트)
  ├── mul_136_kar (136×136 → 272비트, 카라추바)
  │     ├── mul_68_kar[0] (68×68 → 136비트)  ← A₁×B₁
  │     │     ├── mul_36[0] (34×34)  ← A₁₁×B₁₁
  │     │     │     └── mult_gen_0 × 4
  │     │     ├── mul_36[1]  ← A₁₀×B₁₀
  │     │     │     └── mult_gen_0 × 4
  │     │     └── mul_36[2]  ← (A₁₁+A₁₀)×(B₁₁+B₁₀)
  │     │           └── mult_gen_0 × 4
  │     ├── mul_68_kar[1]  ← A₀×B₀
  │     │     └── (동일 구조: mul_36 × 3, mult_gen_0 × 12)
  │     └── mul_68_kar[2]  ← (A₁+A₀)×(B₁+B₀)
  │           └── (동일 구조: mul_36 × 3, mult_gen_0 × 12)
  │
  └── mod_red_1305 (261비트 → 130비트, 6스테이지)
```

#### 4.2.1 병렬 곱셈기 수 (카라추바, 단일 mul136_mod_red당)

| 레벨 | 모듈 | 인스턴스 수/부모 | 총 병렬 인스턴스 |
|------|------|-----------------|------------------|
| 1 | mult_gen_0 (17×17) | 4/mul_36 | **36** |
| 2 | mul_36 (34×34) | 3/mul_68_kar | **9** |
| 3 | mul_68_kar (68×68) | 3/mul_136_kar | **3** |
| 4 | mul_136_kar (136×136) | 1/mul136_mod_red | **1** |

**한 번의 136×136 곱셈에 36개의 17×17 곱셈기가 동시에 작동한다. (기본 대비 56%)**

### 4.3 카라추바 알고리즘이 곱셈기를 줄이는 원리

**표준 교과서 곱셈 (4개 부분곱):**

```
A = A₁ × 2^n + A₀
B = B₁ × 2^n + B₀

A × B = A₁B₁ × 2^{2n} + (A₁B₀ + A₀B₁) × 2^n + A₀B₀
      = M₃ × 2^{2n} + (M₁ + M₂) × 2^n + M₀

필요한 곱셈: 4개 (M₀, M₁, M₂, M₃)
```

**카라추바 곱셈 (3개 부분곱):**

```
M₀ = A₀ × B₀
M₁ = A₁ × B₁
M₂ = (A₀ + A₁) × (B₀ + B₁)

A × B = M₁ × 2^{2n} + (M₂ - M₁ - M₀) × 2^n + M₀

필요한 곱셈: 3개 (M₀, M₁, M₂)
추가 비용: 가산 2회, 감산 2회
```

**저축률:** 각 레벨에서 곱셈 수가 4 → 3 (25% 감소). 두 레벨(136→68→34)에 적용하면:
- 기본: 4 × 4 = 16 mul_36
- 카라추바: 3 × 3 = 9 mul_36
- **절감: 9/16 = 56.25% (43.75% 절감)**

### 4.4 전체 Poly1305 파이프라인의 병렬 곱셈기 배치

#### r_power_n의 6개 사전 계산 곱셈기

```
시간 →     0        18       36       54       72       90      108
           │         │        │        │        │        │        │
r×r ───┬──[mul_0]──→ r²      │        │        │        │        │
       │             │        │        │        │        │        │
       └── r²×r² ───[mul_1]──→ r⁴     │        │        │        │
                               │       │        │        │        │
                               └── r⁴×r⁴ ──[mul_2]──→ r⁸       │
                                               │        │        │
                                               └── r⁸×r⁸ ──[mul_3]──→ r¹⁶
                                                                │        │
                                                                └── [mul_4]──→ r³²
                                                                         │
                                                                         └── [mul_5]──→ r⁶⁴
```

6개의 mul136_mod_red가 직렬 체인이지만, **파이프라인 특성으로 인해 이전 곱셈이 완료되기 전에 다음 곱셈이 시작**된다. 즉:
- 클럭 0: mul_0이 r×r 계산 시작
- 클럭 18: mul_0 완료, r² 출력 → mul_1이 r²×r² 계산 시작
- 클럭 36: mul_1 완료, r⁴ 출력 → mul_2 시작
- ...
- 클럭 108: mul_5 완료, r⁶⁴ 출력

#### mul_red_pipeline의 5개 곱셈기

```
시간 →     108      126      144      162
           │         │        │        │
u1: A×B ──[mul]──→ u1_out    │        │
u2: C×D ──[mul]──→ u2_out    │        │     ← u1, u2, u3 동시 시작
u3: E×F ──[mul]──→ u3_out → [shift_reg 18단]
                    │                  │
                    └── u4: u1×u2 ────[mul]──→ u4_out
                                               │
                                               └── u5: u4×u3 ──[mul]──→ data_out
```

**동시 활성 곱셈기 수:**
- 시간 108~126: u1, u2, u3 = **3개** 동시
- 시간 126~144: u4 = **1개**
- 시간 144~162: u5 = **1개**

#### Poly_1305_pipe의 1개 누적 곱셈기

매 클럭마다 새 블록을 수용하여 cᵢ × r^(n-i+1)을 계산한다.

## 6. 모듈러 리덕션 (mod 2^130-5)

### 6.1 수학적 원리

p = 2^130 - 5에서:

```
2^130 ≡ 5 (mod p)
```

따라서 임의의 x에 대해:

```
x = x_H × 2^130 + x_L
x mod p = (x_H × 5 + x_L) mod p
```

여전히 130비트를 초과할 수 있으므로 반복 적용한다.

### 6.2 mod_red_1305.vhd 6-스테이지 파이프라인

```vhdl
-- 입력: data_in(260 downto 0) = 261비트

-- 스테이지 1: 상위 비트 추출 및 ×5 계산
data_in_1 <= data_in(260 downto 130);           -- 상위 131비트
data_in_2 <= data_in(260 downto 130) & "00";    -- 상위 × 4
B_1 <= ('0' & data_in_2) + data_in_1;           -- 상위 × 5

-- 스테이지 2: A₁ + B₁
A_1 <= data_in(129 downto 0);                   -- 하위 130비트
res_1 <= B_1 + A_1;                             -- 상위×5 + 하위

-- 스테이지 3: 오버플로우 처리
A_2 <= res_1(129 downto 0);                     -- 하위 130비트
B_2 <= res_1(134 downto 130) × 5;              -- 5비트 오버플로우 × 5

-- 스테이지 4: 가산
res_2 <= A_2 + B_2;

-- 스테이지 5: 최종 오버플로우 처리
A_3 <= res_2(129 downto 0);
B_3 <= res_2(130) × 5;                         -- 1비트 오버플로우 × 5

-- 스테이지 6: 최종 결과
res_3 <= A_3 + B_3;
data_out <= res_3(129 downto 0);               -- 130비트 결과
```

**왜 3번 반복하는가?**
- 1차 리덕션 (스테이지 1-2): 261비트 → ~135비트
- 2차 리덕션 (스테이지 3-4): ~135비트 → ~131비트
- 3차 리덕션 (스테이지 5-6): ~131비트 → 130비트 (보장)

### 6.3 ×5 연산의 효율적 구현

```vhdl
x × 5 = x × 4 + x × 1 = (x << 2) + x
```

곱셈 없이 2비트 시프트와 가산만으로 구현한다. FPGA에서 매우 효율적이다.

## 7. Poly1305 파이프라인 누적기

### 7.1 Poly_1305_pipe.vhd 동작

```
블록 n 수신 → [× r^n] → 누적기(acc)에 가산
블록 n-1 수신 → [× r^(n-1)] → 누적기에 가산
...
블록 1 수신 → [× r¹] → 누적기에 가산
→ 모듈러 리덕션 → + s → 태그 출력
```

### 7.2 누적 과정 상세

```vhdl
-- mul_in_A, mul_in_B 준비
res_1 <= '0' & acc(130 downto 0) + ('0' & "00" & Blck);  -- acc + block
mul_in_A <= "000000" & res_1(129 downto 0);               -- 136비트로 패딩
mul_in_B <= "000000" & r;                                  -- r^n (130비트를 136비트로 패딩)

-- mul136_mod_red로 곱셈+리덕션
u1: mul136_mod_red port map(clk, mul_in_A, mul_in_B, mul_out);

-- 누적
process(clk)
begin
    if rising_edge(clk) then
        if shreg_tvalid(18) = '1' then
            if shreg_tlast(18) = '1' then
                acc <= (others => '0');    -- 리셋
            else
                acc <= acc + mul_out_1;    -- 누적
            end if;
        end if;
    end if;
end process;
```

### 7.3 최종 태그 생성

```vhdl
-- 누적 결과의 모듈러 리덕션
acc_res <= acc + mul_out_1;
mod_red_in <= acc_res(260 downto 0);
u2: mod_red_1305 port map(clk, mod_red_in, mod_red_out);

-- s 가산으로 최종 태그 생성
process(clk)
begin
    if rising_edge(clk) then
        if shreg_tlast(25) = '1' then
            tag <= '0' & mod_red_out + ('0' & "00" & shreg_s2(5));
        end if;
    end if;
end process;

-- 바이트 순서 반전 후 출력
data_out <= order_128(tag(127 downto 0));  -- 128비트 태그
```

## 9. r 클램핑 메커니즘

### 9.1 RFC 7539 클램핑 요구사항

r 값의 특정 비트를 0으로 마스킹하여 보안을 강화한다:

```
r을 4개의 32비트 리틀 엔디안 워드로 해석:
  r[0]: 비트 [3:0]의 상위 비트 클리어 → AND 0x0fffffff
  r[1]: 비트 [1:0] 클리어 → AND 0x0ffffffc
  r[2]: 비트 [1:0] 클리어 → AND 0x0ffffffc
  r[3]: 비트 [1:0] 클리어 → AND 0x0ffffffc
```

### 9.2 VHDL 구현

```vhdl
r_clmp <= order_128(r) AND x"0ffffffc0ffffffc0ffffffc0fffffff";
```

`order_128`은 128비트 값의 바이트 순서를 반전하는 함수이다.

### 9.3 클램핑이 적용되는 위치

- **기본 Poly1305.vhd:** 모듈 내부에서 클램핑
- **파이프라인 구현:** AEAD_ChaCha_Poly.vhd (또는 AEAD_encryptor_kar.vhd)에서 Poly1305 파이프라인에 전달하기 전에 클램핑

```vhdl
-- AEAD_ChaCha_Poly.vhd / AEAD_encryptor_kar.vhd
r_ordered <= r AND x"0ffffffc0ffffffc0ffffffc0fffffff";
s_ordered <= s;  -- s는 클램핑 없음
```

## 10. 전체 인스턴스 수 및 자원 분석

### 10.1 기본 구현 (src/) mult_gen_0 총 인스턴스 수

```
Poly_1305_pipe_top 계층:
  ├── r_power_n
  │     ├── 6 × mul136_mod_red = 6 × 64 mult_gen_0 = 384
  │     └── mul_red_pipeline
  │           └── 5 × mul136_mod_red = 5 × 64 = 320
  │
  └── Poly_1305_pipe
        ├── 1 × mul136_mod_red = 1 × 64 = 64
        └── 1 × mod_red_1305 (곱셈기 없음)

총 mult_gen_0: 384 + 320 + 64 = 768개
```

### 10.2 카라추바 최적화 (src_dsp_opt/) mult_gen_0 총 인스턴스 수

```
Poly_1305_pipe_top_kar 계층:
  ├── r_power_n_kar
  │     ├── 6 × mul136_mod_red(kar) = 6 × 36 = 216
  │     └── mul_red_pipeline
  │           └── 5 × mul136_mod_red(kar) = 5 × 36 = 180
  │
  └── Poly_1305_pipe_kar
        ├── 1 × mul136_mod_red(kar) = 1 × 36 = 36
        └── 1 × mod_red_1305 (곱셈기 없음)

총 mult_gen_0: 216 + 180 + 36 = 432개
```

### 10.3 예상 DSP48 사용량

각 mult_gen_0 (17×17 곱셈)은 Xilinx FPGA에서 **1개의 DSP48** 블록에 매핑된다:

| 구현 | mult_gen_0 수 | 예상 DSP48 수 |
|------|-------------|-------------|
| 기본 (Poly1305만) | 768 | ~768 |
| 카라추바 (Poly1305만) | 432 | ~432 |

**참고:** 전체 AEAD 설계에는 Poly1305 외에 ChaCha20도 포함되나, ChaCha20은 곱셈을 사용하지 않으므로 DSP 사용량은 Poly1305에서만 발생한다.

### 10.4 시프트 레지스터 메모리 요구량

| 위치 | 기본 (비트) | 카라추바 (비트) |
|------|-----------|---------------|
| r_power_n 시프트 레지스터 | ~49,860 | ~53,000 |
| mul_red_pipeline | 2,340 | 2,340 |
| Poly_1305_pipe | ~7,000 | ~7,000 |
| Poly_1305_pipe_top | ~42,600 | ~44,700 |
| **총계** | **~101,800** | **~107,040** |

이 시프트 레지스터들은 FPGA에서 분산 RAM(LUTRAM) 또는 블록 RAM(BRAM)으로 구현된다.

### 11.8 mult_gen_0.vhd (17×17 원시 곱셈기)

**엔티티 포트:**
```
CLK : in  STD_LOGIC
A   : in  STD_LOGIC_VECTOR(16 downto 0)  -- 17비트 입력 A
B   : in  STD_LOGIC_VECTOR(16 downto 0)  -- 17비트 입력 B
P   : out STD_LOGIC_VECTOR(33 downto 0)  -- 34비트 출력
```

**내부 (3-스테이지 파이프라인):**
```
sum0 ← unsigned(A) × unsigned(B)         -- 스테이지 1: 곱셈
sum1 ← sum0                               -- 스테이지 2: 레지스터
sum2 ← sum1                               -- 스테이지 3: 레지스터
P    ← std_logic_vector(sum2)             -- 출력
```

이 3단 파이프라인은 FPGA DSP48 블록의 내부 레지스터(A/B 레지스터 + M 레지스터 + P 레지스터)에 자연스럽게 매핑된다.
